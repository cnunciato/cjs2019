Infrastructure as Code

  • the concept of "infrastructure as code."
  • Sounded buzzwordy and vague to me at the time, but really is about something very specific and kind of at the core of modern software automation and delivery today.
  • So if by "infrastructure", we mean this stuff →
  • -- the servers, networks, storage, DNS, higher-level services, and so on -- the stuff our apps need to survive...
  • then "infrastructure as code" is just the representation of all (or most) of that stuff as code →
  • like this, so that we can manage it much like we've learned how to manage our software.
  • Why would we do this? Well, the journey sometimes starts out a little like this →
  • This is me, clicking around in the D.O. dashboard, creating a virtual machine and a database in order to stand up a typical web application, Googling around to figure out how to install some stuff. Just doing the thing.
  • Totally works, took about a half-hour to stand up this app, and really wasn't too bad, and at the end of it, I had a working thing.
  • But this approach brings with it a few problems. →

üò¨

Can't see what's inside.

Can't duplicate it.

Can't scale it out easily.

Can't recover from disaster.

  • For one, you can't see what's inside -- you can't look at this server from the outside and see what's on it, what's installed, what version of an app we're running, and so on.
  • You can't duplicate it -- it's a "unique snowflake". If you had to make another one of these, you'd have to do this all this stuff all over again -- not terrible, but not a party, either.
  • You can't scale it out. If this app took off, and you needed like five or ten more instances of it, you've got a little work to do -- and then anything that needs doing on one of them (bumping Node, releasing an app), needs doing on all of them, and since we're human, we're likely to mess that up once in a while, too.
  • And you can't recover from disaster. Things go wrong. It is super simple to delete the wrong thing in a dashboard, and if those machines all disappeared and couldn't be built up from scratch easily →
  • ... you're in for a sad, sad time.
  • This is my daughter. I got her permission.
  • Infrastructure as code fixes these problems by taking our hands out of the computers →
  • ... and putting them into our IDEs, where we can write code that expresses what we want, and let the tools run that code to figure out how to make it happen.
  • Cool. So what's it look like?

  • Configuration management.

  • We'll take a quick look at both.
  • This is Chef. Chef is a configuration management tool.
  • This is a Chef recipe that installs the Nginx web server, sets a default document for it, and starts it.
  • When I run Chef with this recipe and point it at a server somewhere, Chef will "make it so" -- it will drive that server to the end state I've declared here.
  • If I run it again, nothing will happen, because that desired state will have been achieved.
  • Config management tools generally focus on taking existing infrastructure, configuring it -- installing software, creating users and files and things -- and keeping it updated over time.

‚úÖ Commonalities

They're declarative: We express the what, and the tools determine the how.

They're idempotent: Given an input, we get the same result, however many times we run it.

  • These tools generally have two things in common.
  • They're declarative: We express the what, and the tools determine the how. Instead of saying do this, then this, then this, we say, I want this, and the tools go off and do it.
  • They're idempotent, which is a term borrowed from functional programming that here means, Given some code as input, we can run that code over and over again and get the same result.
  • So -- what all this stuff gives us is infrastructure that's more reliable, repeatable, replicable, testable, self-documenting and collaborative, and that just leads to less sadness and more happiness.
  • Takes a bit of learning, bit of work, but it's super worth it, because it lets us ship lots of software, really often, really safely, and that's kind of what it's all about.
  • And also, it just feels awesome to be able to write code that makes and destroys computers. That's just fun. For some.

  • I've thought about this a lot, and I don't think it's that we don't care about infrastructure.
  • I think we'd like to be able to manage this stuff if we could.
  • I just think we haven't been shown that we can do it, with tools that extend our mental models and that fit into our existing workflows.
  • These tools are amazing, they really are -- but they are not made for developers: they're made for managing the operating environment, so their abstractions kind of end there.
  • WE, as developers, tend to cruise at a much higher altitude. That's my app, in that box there, and I just want my box to work. I know I want Node 10, maybe, but below that, I'm in unfamiliar territory.
  • If we're going to manage our own infrastructure, we really need the tools and the abstractions to meet us where we are. Or we just won't use them.
  • And I think that's why we haven't.
  • But over the last year or two, some interesting things have been happening.
  • One was that some very developer-focused developers, from outside the ops world, recognized this problem and went to work on bringing modern languages -- LIKE JAVASCRIPT -- to the infrastructure party.
  • Another was that cloud providers like AWS began to move beyond these rather heavyweight constructs like VIRTUAL MACHINE and DATABASE CLUSTER into much more lightweight, application-friendly constructs like FUNCTION, EVENT, ROUTE, TABLE, RECORD...
  • Things that fit much more naturally into application code.
  • So it turns out that when you combine the practice of infrastructure as code, and all we've learned there
  • with the expressive and compositional power of a modern programming language like JavaScript
  • AND these now very lightweight, app-friendly cloud primitives...
  • What you get is something we've never really had before.
